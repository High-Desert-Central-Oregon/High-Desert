Orientation

This is the how, concretely, in order — the layer below the dev framework. Everything here implements a pattern; the ties are marked so the reasoning stays attached to the code.

What the prototype is

A functional MVP a ~50-person Redmond founding cohort can actually use during the closed beta — real verification, real neighborhood events, a real governance vote, a real public transparency view. It is the smallest thing that is unmistakably High Desert (Pattern 17, Minimum Trustworthy Platform).

It is not the full platform, and not a design mockup. If you also want something to show Compass and the cohort before code exists, a clickable Figma flow is a cheap parallel track — but the spec below is the buildable thing.

Each has a recommended default so you can proceed today; all are flagged again in §07.

unblocks Verification (A1) · Patterns 1, 2, 24

Verification pathways for v1

Recommend a multi-path, human-reviewed flow: any one of — driver's license / state ID, utility bill, voter registration, property record, or a mailed postcard with a one-time code. The postcard path solves PO-Box-only delivery (Warm Springs) and the unbanked/undocumented case. A person clears every submission from a review queue; evidence is deleted the moment a decision is made.

unblocks Neighborhoods (B2) · Pattern 4

How a neighborhood is defined

Locked: resident-set, from a seeded list. Seeded with the 35 Redmond neighborhoods in §03 (sourced from the Enjoy Bend Life Redmond neighborhood map), picked by the member on join and editable by community process later — never a radius an engineer guessed. One entry carries a flag for your call (see §03): eagle-crest (a resort west of the city limits).

unblocks Governance (A2) · Patterns 5, 6

Provisional governance numbers

Stub real values in config: quorum 20% of eligible voters; minor = simple majority; major = 60%; immutable = 75% + 30-day window. Tenure weight: Yr1 1.0× · Yr2 1.2× · Yr3+ 1.5×. These are placeholders the community ratifies — but the code needs numbers now.

design integrity · Pattern 6 (anti-coercion)

Vote visibility

Recommend secret until close, aggregate-only after. Individual ballots are never readable by other members; only the tallied result and the audit metadata are public. Prevents pressure and vote-buying.

02

Prototype Scope

The line between the First Build Slice and everything that comes after. Ruthless scope is the point — feature creep before trust is the documented common failure (Pattern 17).

In — the prototype builds

  • Auth & member profile — magic-link sign-in, profile row
  • Residency verification — multi-path, human-reviewed, verify-then-forget (A1)
  • Neighborhoods — resident-set from a seeded list (B2)
  • Neighborhood event primitive — create / list / detail / RSVP / who's-bringing (B1, B3)
  • Governance — proposal + tenure-weighted vote + result + audit log (A2)
  • Moderation — report → graduated action → appeal, human decides (A3)
  • Transparency view — public aggregate health + mod log + vote results (A4)
  • Plain-language T&C — versioned, read-confirm, consent record (A5)
  • Accessibility + Spanish + low-bandwidth — cross-cutting, not a later pass

Out — deliberately deferred

  • Mutual aid board (C1)
  • Skills & trades trust graph (C2)
  • Skill exchange / learning webs (C3)
  • Resilience alerts & SMS (C5, Phase 2)
  • Real payments + sliding-scale UI (post-beta)
  • The regional discussion board (deferred / logged)
  • Visitor tier surfaces, youth version, historical archive
  • Multi-community expansion (Redmond only)

Why each is deferred or out

The quick grid above, with the reasoning attached — so the scope line can be re-decided in context, not just obeyed.

deferred — planned, not in the prototype

Coming later, just not first

Mutual aid board (Phase 1) — a participation feature that only works once a critical mass of verified neighbors and a trust culture exist; built early it sits empty or gets misused.

Skills & trades trust graph (Phase 1) — the hardest knowledge feature to get right because it must resist becoming a score (P10); needs mature verification and vouching norms first.

Skill exchange / learning webs (Phase 1) — depends on the trust graph and an active event culture; matching people to learn from each other needs enough verified members to matter.

Standing / recurring gatherings (near-term) — a complexity layer on the event primitive; the single-event MVP must prove people will host and show up before recurrence earns its state.

Community resilience alerts + SMS (Phase 2) — high-stakes (people rely on it in an emergency), so it needs redundant, tested delivery you never bolt onto an MVP; adds SMS cost and a provider.

Historical / community archive (Phase 2, gated) — held until Warm Springs consultation, because cultural memory is sovereignty-sensitive and must not be designed first and asked about later.

Youth version (13–17) (Phase 3, gated) — a real legal/safety surface (COPPA, isolated feed) requiring a community vote before launch.

Real payments + sliding-scale UI (post-beta) — the cohort is free, so charging is unnecessary now; deferring removes the Stripe/PCI surface and lets fieldwork teach the need-distribution before the sliding scale is locked (Meadows).

Multi-community expansion (Year 1+) — Redmond-first is the strategy; saturate and win trust in one community before branching, gated on Redmond being healthy.

Regional discussion board (deferred & logged) — the single highest-conflict, highest-moderation-burden surface and the thing that turned Nextdoor toxic; held until the trust stock is high and re-opened only by a community vote.

out by design — refusal, or another entity's lane

Not building, on purpose

Advertising (permanent, immutable) — a second customer whose interests oppose members' is the root of platform decay; structural, not deferrable (Doctorow, Wu).

Scope-based pricing (decided against) — charging more for a wider area feels fair but inverts fairness (breadth correlates with need → regressive) and installs an enshittification value-withhold; replaced by price-by-means (Baradaran, Doctorow).

Ranking / recommendation / engagement-optimizing algorithms (out) — they sever feedback from the people they affect and become optimization weapons; default is chronological + proximity, any future ranking member-visible and never a target (O'Neil, Christian, P8).

Local marketplace / buy-sell (out — Compass's lane) — commercial discovery belongs to Redmond Compass; HD does non-monetary aid and exchange first, and a transactional market would import the ratings pressure HD refuses.

Visitor posting / voting (out) — full participation requires verified residency (Ostrom's boundary); visitors get read-only public info only.

Data Model

Postgres tables for the slice. Two non-negotiables show up in the schema itself: store as little PII as possible (verify-then-forget, Pattern 2) and votes, moderation, and the audit log are append-only (reversibility & the record, Pattern 18).

04

RLS — the Constitution in Code

The governance model is enforced at the row level, where the data lives — so the UI is never the only thing between a rule and its breach (Pattern 18). A representative table; the principle is the same throughout: verified status and tenure weight are computed by the server and trusted from nowhere else.

TableReadWrite
profilesPublic fields readable by verified membersOwn row only; verified_status/role/tenure_start blocked from self-edit (service role only)
verificationsOwn status; evidence readable only by reviewersInsert own request; status set by reviewers only
events / event_rsvpsVerified membersInsert if verified_status = 'verified'; edit/delete by creator or moderator
proposals / votesProposals: verified. Individual votes: never — aggregate onlyOne vote per proposal while open; weight set server-side
moderation_actions / audit_logPublic (no PII)Insert by moderators/service; no update, no delete (append-only)

Two policies worth seeing in full — the vote-integrity guard and verify-then-forget:

-- A member may cast at most one vote per open proposal.
-- The weight column is set by a trigger from tenure, never by the client.
create policy "one vote per open proposal" on votes
for insert to authenticated
with check (
  exists (select 1 from profiles p
          where p.id = auth.uid() and p.verified_status = 'verified')
  and exists (select 1 from proposals pr
          where pr.id = proposal_id and pr.status = 'open'
            and now() between pr.opens_at and pr.closes_at)
  and not exists (select 1 from votes v
          where v.proposal_id = proposal_id and v.user_id = auth.uid())
);

-- Verify, then forget: on any decision, null the evidence and delete the file.
create trigger drop_evidence_on_decision
after update of status on verifications
for each row when (new.status in ('approved','rejected'))
execute function purge_verification_evidence();  -- removes storage object + nulls path
05

Implementation by Feature

For each in-scope feature: the data it touches, the policy that protects it, the member flow, and the edge cases that actually matter in Central Oregon.

Auth & profile · framework foundation · Pattern 3

Sign-in & member profile

Data: profiles row created by trigger on auth.users insert. Flow: magic-link sign-in → profile created as visitor → must accept T&C and pass verification before any participation unlocks. Edge: a profile with no verified status can browse only; the ladder is enforced in RLS, not just hidden in the UI.

Ties: P3 Participation Ladder
A5 · Pattern 22

Plain-language terms & read-confirm

Data: documents (versioned) + consents. Flow: render the current version, require a real scroll-and-confirm (System-2 friction on purpose), write a consent row bound to that version. Edge: when terms change, members re-confirm the new version; old consents stay on record. A plain-language draft of both documents now exists (companion: Terms of Membership & Privacy Policy v1) to seed these rows — pending Oregon legal review before launch.

Ties: P22 Effort Where It Belongs
A1 · Patterns 1, 2, 24

Residency verification (the trust substrate)

Data: verifications + a private storage bucket. Flow: member picks a path (ID / utility bill / voter reg / property record / mailed postcard code) → uploads or requests a postcard → lands in a reviewer queue → a person approves/rejects → on decision, evidence is purged and only the status flag remains. Edge: the postcard-code path is the PO-Box / Warm Springs / unbanked accommodation and must work on day one; reviewer UI shows the minimum needed to decide and nothing more.

Ties: P1 Verification That Fits the Person · P2 Verify Then Forget · P24 Built for Every Body
B2 · Pattern 4

Neighborhoods

Data: neighborhoods seeded; profiles.neighborhood_id. Flow: on join (post-verification) the member picks their neighborhood from the seeded list. Edge: "none of these fits" routes to a human, not a forced choice; boundaries are editable by community process later, never a hard radius.

Ties: P4 Neighborhoods Drawn by Residents
B1, B3 · Patterns 11, 12

Neighborhood event primitive (the headline)

Data: events + event_rsvps. Flow: a verified member creates a neighborhood-scoped gathering (title, when, where, capacity); others RSVP and add what they're bringing; carpool field optional. Edge: there is no open comment thread — coordination ends when the event does. This is the deliberate refusal of the discussion-board surface.

Ties: P11 Events as Participation · P12 Just Enough Coordination
A2 · Patterns 5, 6

Governance: proposal & tenure-weighted vote

Data: proposals, votes, audit_log. Flow: a member opens a proposal (kind = minor/major/immutable) → it runs for a set window → verified members cast one ballot each → a server function tallies using tenure weights and the kind's threshold → result + metadata written to the audit log. Edge: ballots are secret until close and never individually readable; weight is computed server-side from tenure_start, never accepted from the client. Seed the beta with one small real vote (a naming or norm decision) to prove it works.

Ties: P5 Decide the Rules Before the Tool · P6 Tenure-Weighted Vote
A3 · Patterns 7, 9, 19

Moderation & appeal

Data: moderation_actions (append-only), appeals. Flow: a member reports content → a human moderator takes a graduated action (warn → temp → extended → review) → the appeal opens immediately in the same view. Edge: no permanent ban without review; a person always makes the call — automation may flag, never decide; sustained complaints against a decision can trigger a community override vote (reuses the governance engine).

Ties: P7 Action, Then Voice · P9 The Override · P19 Human in the Loop
A4 · Pattern 8

Transparency view

Data: read-only aggregates over audit_log, moderation_actions, votes. Flow: a public page shows member counts, events held, the moderation-action log (no PII), and closed-vote results. Edge: these are vital signs, not targets — displayed for accountability, never optimized, never shown as per-member scores. Quietly compute exit-vs-appeal for your own eyes as the early-warning signal.

Ties: P8 Vital Signs, Not Targets
cross-cutting · Patterns 24, 22 · design practice

Accessibility, Spanish & low-bandwidth

How: semantic HTML, full keyboard paths, real contrast, captions, and an i18n string layer (English + Spanish together) from the first screen; test on a deliberately slow phone. Edge: not a later mode — built in. Spanish parity matters most when Madras branches later, but the architecture is laid now so it isn't a rewrite.

Ties: P24 Built for Every Body
06

Build Sequence

Dependency-ordered. The trust substrate is built before anything to participate in; the discussion-style surfaces are never built at all. This maps to the June "technical build" milestone and feeds the closed-beta gate in Business Plan v6 §07.

  1. Scaffold & seed

    Next.js + Tailwind app, Supabase project, env + CI + deploy target. Create the schema and RLS skeleton. Seed neighborhoods and the first T&C document.

    Done when: a deployed empty app talks to the database under RLS.

  2. Auth & profile

    Magic-link sign-in; profile-on-signup trigger; profile RLS (own-edit, server-only status). Visitor can browse, nothing else.

    Gate: status/role/tenure provably not self-editable.

  3. Terms & consent

    Versioned T&C render + scroll-confirm + consent record. Participation stays locked until consent + verification.

    Ties: P22 · A5

  4. Verification + reviewer queue

    Multi-path submission, private evidence bucket, reviewer queue UI, approve/reject, evidence-purge trigger. This is the substrate — it precedes all participation features.

    Gate: postcard-code (PO-Box) path works; evidence is gone after decision.

  5. Neighborhoods

    Self-assign from the seeded list; "none fits" → human. Unlocks scoping for events.

    Ties: P4 · B2

  6. Event primitive

    Create / list / detail / RSVP / bringing, neighborhood-scoped. No comment feed. The headline build.

    Ties: P11, P12 · B1, B3

  7. Governance engine

    Proposal → window → one weighted ballot each → server tally → audit row. Secret-until-close. Run one real seed vote.

    Gate: weight server-set; one-vote-per-proposal enforced in RLS.

  8. Moderation, appeal & transparency

    Report → graduated action (append-only) → immediate appeal; public transparency page over the audit log. Reuses the governance engine for overrides.

    Ties: P7, P8, P9, P19 · A3, A4

  9. Accessibility, Spanish & mobile pass

    Cross-cutting QA: keyboard, screen-reader, contrast, i18n strings, slow-phone. Built along the way; verified here.

    Ties: P24

  10. Closed-beta hardening

    Full RLS review, soft-delete audit, seed the ~50-person cohort, dry-run the first conflict + first vote. Then — and only then — the public-launch gate from the business plan applies.

    Gate: two trained mods · RLS reviewed · the slice works end-to-end.
